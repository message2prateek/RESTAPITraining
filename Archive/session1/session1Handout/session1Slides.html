Client-Server Architecture

To understand how modern Websites and Apps work, we need to understand the concept of front-end vs. back-end.

Waiting at a Restaurant

  • Just like there are waitstaff and kitchen staff in a restaurant, front-end and back-end divide the functionality of your site.

  • It allows each side to do what they are good at.

Front-End

  • Front-end code creates a user interface where a controlled interaction happens.

  • Users need something that they can browse. In a restaurant setting, this is the Menu.

  • From a developer’s perspective, this is similar to HTML and CSS.

API

  • We need to communicate orders to the kitchen staff.

  • Waiters and waitresses can help you understand the menu and then take your orders to the kitchen staff.

  • Waiters here are the API, they are hiding away the complexity of the kitchen and cooking from the customer.

  • JavaScript is used for sending user requests to the back-end, by invoking the APIs.

Waiters hide away the complexity of the kitchen from the customer and save the customer from the Chef and visa-versa.

Back-End

  • The waitstaff and menus provide a friendly, perfectly-formatted version of what goes on in the kitchen, without revealing any of the processes.

  • This is similar to the back-end of your web application, or the code that runs on the server-side.

  • Just like the kitchen, the server is in a different location than the user interface.

  • The server must send a response whenever it receives a request through the API.

  • In the case of the restaurant, this response could be:

    1. The food
    2. A notice that the kitchen has run out of stock for that meal.
    3. A follow-up question that the waiter did not ask.

Why we need front-end, backend, and the API?

  • Can you imagine if the chef had to take the place of the waiters? It would be a customer experience disaster!

  • Separation of duties between Front-end and Back-end allows each side to focus on the challenges that they are uniquely qualified to handle.

Understanding REST APIs using Bank Analogy

Bank of Trust

In our Open Access bank, for a customer to deposit money, they will have to:

  1. Go to the bank with the money in their hands;
  2. Open the vault and put the money inside;
  3. Go to the logbook and write “I am <name> and I deposited <amount> into the vault.”;
  4. Leave the bank.

There are two problems with this approach:

  1. The vault door is very hard to open and requires physical strength.

  2. In this model, we are trusting that people will be absolutely honest.

Enters Bank Teller

  • What if we hire a person, who’s strong and smart.

  • It is no longer possible for a customer to directly interact with the vault.

In our new bank, for a customer to deposit money, they have to:

  1. Go to the bank with the money in their hands;

  2. Approach the bank teller Sally and say: “I am <name> and would like to deposit <amount> into my account.”;

  3. Sally takes the money, opens the vault and puts the money in. She is very strong and would be able to do this with no problem.

  4. Sally records this transaction on the log book.

  5. Sally goes back to the customer and tells them that the money has been deposited.

  6. The customer leaves the bank.

Now nobody needs to open that bulky vault door anymore, and the accessibility office is pleased with the improvement.

API == Application Programming Interface

  1. APIs allow you to talk to other software easily.

  2. APIs allow us to swap components as long as we follow the same protocol.

Parts of BANK

Interface

You might have noticed that there are two parts of the bank, and there are two areas:

  • The front area, where customers queue up in an orderly fashion;

  • and the back area, where the money handling happens.

  • In the middle, we have the area where the front meets the back. Neither can cross the other side and interactions can only happen in this area.

In API Terminology, we call this middle area the Interface; it is a place where software components interact.

In our bank, the front area and the back area both agree to use this interface to exchange information.

Protocol

But what if Bob asked, “Could you give me some information about underwater basket weaving?”

In API Terminology, a Protocol is a set of rules defining how components can interact with each other.

Format

  1. The language of exchange so far has been English. We call this a Format.

  2. The format specifies how you can encode the data you want to send over to the other party.

  3. In the real world, common formats for web APIs include JSON and XML.

Endpoint

Finally, let’s assume we want to expand our business into stock markets as well; we need a special kind of bank teller to deal with stock market trading.

In API Terminology, an API Endpoints refers to the different functionalities that the API provides within the same Interface.

In this case, Tom and Sally are both endpoints.

To sum up

Interface is a place where different software components interact.

Protocol is a set of rules defining how they interact.

Format defines how they talk to each other.

Endpoints provide different functionalities within the same interface.

What else can Bank Tellers do

Verification

Sally can verify that Bob has sufficient balance when he tries to withdraw $10,000.

All APIs should verify information and not depend on front-end.

Error Reporting

Sally can nicely tell Bob that he does not have sufficient balance.

APIs can have error reporting mechanisms to indicate an error has occurred, saving us loads of head scratching when things go south.

Pagination

Problem
Bob wants to check his account info. Trouble is that he has 200 accounts.

Solution
  • Sally can provide information on 10 accounts at a time.
  • When Bob is finished with 10, he can then move on to the next 10.
  • This is called pagination

Pagination save us bandwidth and server resources as we don’t need to fetch everything in the dataset at once.

e.g. www.bank.com/accounts?size=10&page=2

Filtering

Problem
Bob can’t recall the account name and only remembers that it started with the letter “A”.

Solution
Bob can ask Sally to only show him accounts starting with the letter “A”. This is called filtering.

Filtering also helps save bandwidth and resources, as well as making it easier to navigate.

e.g. www.bank.com/accounts?nameStarting=A

Access Control

Problem
Bob tries to masquerade as Alice and tries to withdraw $1,000,000 from her account.

Solution
Sally can check for a valid ID and a bank card before she carries out the operation.

Authorisation and Access Control can be built into an API to ensure only authorized personnel to have access to specific data.

Rate limits

Problem

Bob chooses to make multiple withdraws for $0.01 in order to spite Sally.

Solution

The bank could impose a 1-withdrawal-per-10-minutes rule;

If Bob was to continue withdrawing $0.01 each time, he would be here for quite a while.

Representational State Transfer Application Interface (REST)

The boring part.

REST API == REpresentational State Transfer + Application Programming Interface

  • REST stands for REpresentational State Transfer.

  • It is an architectural style using which we can design web services.

    • REST defines few constraints, any API which meets these constraints is called a REST/RESTful API.

    • It is not a protocol.

  • The underlying protocol for REST is HTTP, which is the basic web protocol used for the exchange of data.

How REST API works

The big picture

Server
Server is a big computer. Think ‘n’ times more powerful than your laptop.

API
API is a program that runs on the server. API interfaces with and handles client requests.
Resource :The information we are interested in
Representation
We can’t transfer the physical database to the client, we can only transfer a representation of the information stored on the database, in the format that the client understands.
State
The data in the database might not be static and keep changing. So state is the state of the resource at the time client made the request.

Guiding Principles of REST

  1. Client-Server architecture
  2. Stateless
  3. Cacheable
  4. Uniform interface
    1. Offer access using resources
    2. Represent resources by representations
    3. Exchange self-descriptive messages
    4. HATEOAS
  5. Layered system
  6. Code On Demand (Optional)

1. Client-Server architecture

  • The server does not know anything about the user interface and the client does not know anything about the data storage and retrieval.

  • This separation of duty reduces complexity for both the client and server code.

  • Since the server does not know anything about the user interface we can plug different clients to the same server.

    e.g. our BNZ Internet Banking Website and Mobile Apps(iOS and Android) are different clients which are served by the same server through common APIs.

2. Stateless

  • Each request from the client to server must contain all of the information necessary to understand the request, and cannot take advantage of any stored context on the server.

  • Session state is therefore kept entirely on the client.

Suppose the server serves pages of the book, one page at a time. There are a few ways a client can make the request.

1st method (non-RESTful way) - Client says to the server “Dude, give me next page or bla.com/customers?page=nextPage”. The server now has to know current page client is on i.e. the state.

2nd method (RESTful way) - Client says to server “Dude, give me page number 10 bla.com/customers?page=10”. Here, the client’s request contains all the necessary information that’s needed.

3. Cacheable

  • A REST API should be designed to encourage the storage of cacheable data.

  • This means that when data is cacheable, make a local copy of it.

  • The response should indicate if the data can be cached or not(if real-time data is required).

  • With caching we can greatly reduce the number of interactions with our API, reducing server load.

4. Uniform interface

  • With a uniform interface, the overall system architecture is simplified and the visibility of interactions is improved.

  • REST defines four constraints to ensure a uniform interface.

The interface defines what requests we can make and the format of those requests.

e.g. If we go to the bank to deposit money, we need to fill in a deposit slip. These deposit slips are similar between banks and we know what to expect.

Depositing money will be a whole lot frustrating if deposit slips differed drastically between banks.

4.1 Offer access using resources

  • The key abstraction of information in REST is a resource.

  • Any information that can be named can be a resource: a document or image, a temporal service, a collection of other resources, a non-virtual object (e.g. a person), and so on.

  • REST uses a URI’s to identify the particular resource involved in an interaction between components.

📝 Think of URIs as navigating through a folder structure on your PC. e.g. /customers/1234/accounts/4

4.2 Represent resources by representations

  • A resource is an information which is stored away in a database which the client request from a server.

  • The server cannot transfer the database to the client, it can only give the client a representation of the resource.

  • Now resource can be described/represented in different format like HTML, JSON, XML etc.

e.g. client request to the server is something like

Client: Hey Mr. Server, I need the account number 4 identified by /customers/1234/accounts/4 and please give it to me in JSON format.

Server: Ok, here you go!

{
  "account-number": "4",
  "type": "savings",
  "balance": 1,000,000.00
}

4.3 Exchange self-descriptive messages

In a REST system, we should be able to interpret any message without having seen the previous one.
todolists/7/items/35 means 35th Item from 7th TodoList

Each message includes enough information to describe how to process the message.
If the server returns JSON it should mention that it’s JSON using the media type application/json

Responses also explicitly indicate their cache-ability
Responses should use appropriate headers to indicate if a response could be cached or not. e.g. Cache-Control: no-cache

4.4 Hypermedia As the Engine Of Application State(HATEOAS)

  • We are able to navigate through a website we have never seen before by using links. Links take us to new pages without us needing to manually enter the URL to each page ourselves.

  • The idea here is that the user of a REST API is able to navigate the API without needing to read whole bunch of documentation.

📝 Hypermedia refers to hyperlinks btw.

e.g. The call to customers\1234 returns a response like below

  {
    "customerNumber" : "1234",
    "accounts": ["4", "5"]
  }

It is clear that the customer has two accounts, but how to get to the account information?

However, if the server returns a message like

{
  "customerNumber": "1234",
  "items": ["/customers/1234/accounts/4", "/customers/1234/accounts/5"]
}

we are now able to follow the links in the response to reach the individual item of the Todo list.

5. Layered system

  • The layered system style allows an API to be composed of hierarchical layers(components) such that each component cannot “see” beyond the immediate layer with which they are interacting.

  • The benefit of this is that each component is less complex and can be worked on without affecting other components, thus increasing flexibility and scalability.

6. Code on demand (optional)

  • REST allows client functionality to be extended by allowing scripts/applets to be transmitted over the APIs.

  • The client is then able to execute these scripts/applets thus reducing the number of features required to be pre-implemented on the client.
