HTTP Protocol





📝 REST is not limited to HTTP but is most often implemented using HTTP.

What is a protocol

Making a phone call

There are certain etiquettes we should follow over a phone call.

For example, we know that when the other person is speaking we should be quiet.

Computer communication etiquettes are termed Protocol.

📝 A computer protocol is an accepted set of rules that govern how two computers can speak to each other.

While people are able to make out that "My favorite color is blue" and "Blue is my favorite color" mean the same thing, computers are not that smart.

Computers need exact instructions, therefore computer protocols are pretty strict.

The server has to know in advance what the client request will look like.

Protocol of the web

  • On the web, the main protocol is the Hyper-Text Transfer Protocol, better known by its acronym, HTTP.

  • When we type an address like http://example.com into a web browser, the "http" part of URI tells the browser to use the rules of HTTP when talking with the server.

HTTP Requests

  • Communication in HTTP centers around a concept called the Request-Response Cycle.

  • The client sends the server a request to do something. The server, in turn, sends the client a response saying whether or not the server could do what the client asked.

To make a valid request, the client needs to include four things:

  1. URL (Uniform Resource Locator)
  2. Method
  3. List of Headers
  4. Body


URL (Uniform Resource Locator)

  • A URL is the unique address of the thing(data) called the Resource we are trying to locate on the internet.

  • A URL also tells us the mechanism/protocol to use for reaching that resource e.g. HTTP, HTTPS, FTP etc.

  • This resource can be an HTML page, Image file or raw data such as a record of Person's details etc.



URL Description
Uniform All URLs follow the same structure
Resource URLs locate resources(item/s)
Locator Specifies the full path to resource over the network

HTTP Methods

  • A method describes the action that the client wants the server to take on the resource located by the URI.


The four methods most commonly used with REST APIs are:

  • GET - Asks the server to retrieve a resource
  • POST - Asks the server to create a new resource
  • PUT - Asks the server to edit/update an existing resource
  • DELETE - Asks the server to delete a resource

Ordering a Pizza

As an analogy on how HTTP methods work, we are going to see how we might use the API of a Pizza Store.

Here Pizza is the "thing" we are interested in so it's our resource.

  1. First, we'll send a POST request to restaurant's server to create a new pizza for us.

  2. Once the order is placed we remember that we ordered the wrong topping, so we'll send a PUT request to the restaurant to edit/change the order.

  3. While are wait for our order, we'll send some GET requests to get the status of our pizza.

  4. After an hour of waiting, we finally decide to cancel the order by sending a DELETE request.

HTTP Headers

  • Headers provide meta-data about the request i.e. more information around the request.

  • The information headers provide could be the time the client made the request, the size of the request body etc.

📝 The client can send the User-Agent header to the server to let the server know whether the request is coming from a laptop or a phone, the server can then send back the platform-specific website.

📝 The client can set the Expires header to the client to let the client know when the cache expires.

Body

  • The request body contains the data the client wants to send the server.

  • Continuing our pizza ordering example above, the body is where the order details go.

HTTP Responses

  • HTTP responses have a very similar structure to requests.

  • The main difference is that instead of a method and a URL, the response includes a status code.

  • Beyond that, the response headers and body follow the same format as requests.

Status Codes

  • Status codes are three-digit numbers that each have a unique meaning.
  • For example, you may have seen this page.
  • A return code of 404 means that the resource requested by the client could not be found by the server.
  • There are other codes as well like 200. Which means "success"

Hands-On

Resource Naming

  • A RESTful API ends up being simply a collection of URIs.

  • Each resource has its own address or URI—every interesting piece of information the server can provide is exposed as a resource.

  • The constraint of uniform interface is partially addressed by the combination of URIs and HTTP verbs, and using them in line with the standards and conventions.

Endpoints

In a typical REST API, a resource will have two URL patterns assigned to it.

  • A plural of the resource, like /orders.
  • Single resource, like /orders/<order_id>, where <order_id> is the unique identifier for an order.

These two URL patterns used to identify resource/s are called Endpoints of our API.

📝 These are called endpoints because they go at the end of the URL, as in http://example.com/<endpoint_goes_here>


The resource could be a single resource or a collection, but they are regarded as a single resource.

API Actions

Actions that our API can perform are described by a combination of an HTTP verb and the Endpoint.

Plural Endpoint : Listing existing resources or Creating a new one

Plural Endpoint + Unique Identifier : Retrieving, updating, or canceling a specific resource.

HTTP Verb Endpoint Action
GET /orders List existing orders
POST /orders Place a new order
GET /orders/1 Get details for order #1
GET /orders/2 Get details for order #2
PUT /orders/1 Update order #1
DELETE /orders/1 Cancel order #1

Common scenarios and respective status codes

HTTP Method CRUD Operation Entire Collection (e.g. /orders) Specific Item (e.g. /orders/{id})
POST Create 201 (Created), 409 (Conflict) if resource already exists. 405 (Method Not Allowed)
GET Read 200 (OK), list of customers. 200 (OK), single customer. 404 (Not Found), if ID not found or invalid.
PUT Update 405 (Method Not Allowed), unless entire collection needs to be updated 200 (OK) or 204 (No Content). 404 (Not Found), if ID not found or invalid.
DELETE Delete 405 (Method Not Allowed), unless entire collection is to be deleted 200 (OK). 404 (Not Found), if ID not found or invalid.

Linking Resources Together

  • For our Pizza order resource, the endpoints by convention will be /orders and /orders/<id>

  • But, each order has to be associated with a customer. Customer is another resource and will have its own endpoints. /customers and /customers/<id>

Endpoint Meaning
/customers all the customers
/customers/5 customer with customerId = 5
/customers/5/orders all orders belonging to customer with id =5
/customers/5/orders/3 order with orderId = 3 & belonging to customer with customerId = 5

📝 Id values in the URL are called Path Parameters.

Hands-On

Filtering Data

  • If the dataset is really big then returning all resources at one time is not desired. e.g. GET /accounts

  • We can filter the results by using the Query String component of the URI.

  • Query string is added to the end of the URI separated by a ? e.g. http://example.com/accounts?key=value

  • Multiple query strings can be combined by using & e.g. http://example.com/accounts?key1=value1&key2=value2


Endpoint Meaning
/accounts?status=open All Accounts with status "Open"
/accounts?type=saving All Accounts of type "Savings"
/accounts?status=open&type=savings All Accounts who's status is "Open" AND type = "Savings"

📝 Parameter after the ? is called Query Parameter.

Sorting data

Some endpoints might support returning resources in a sorted order.

e.g. The client can request the server to send the /accounts resources sort by account name.

  • http://example.com/accounts?sort=name (sort by name ascending)
  • http://example.com/accounts?sort=name,desc (sort by name descending)

Pagination

  • If the dataset is too big some endpoints could implement Pagination

  • Pagination means dividing the data into small batches called a Page.

  • e.g. The /accounts data could be split into batches of 100 or 200 accounts.

  • The client can then ask the server to send resources one batch at a time.

  • Page numbers start from Zero. i.e. page=0 is the first page, page=1 is the second page.

Endpoint Page Accounts
/accounts?size=10 first page return first 10 accounts
/accounts?size=10&page=0 first page return first 10 accounts
/accounts?size=10&page=1 second page return accounts 11th-20th
/accounts?size=10&page=2 third page return accounts 21st-30th
